> A B2B marketplace handling split payments, timezone-aware scheduling, and automated booking lifecycle management.

## The Challenge

Corporate event booking involves three-way financial splits (platform, vendor, tax), vendor availability across time zones, and pre-authorized payments that can go stale. Hand-rolled cron jobs for managing thousands of concurrent Stripe pre-authorizations lead to stale locks and failed captures.

## Key Engineering Decisions

### Split Payment Orchestration

**Stripe Connect** manages three-way payment splits between the platform fee, primary vendor, and local tax. Partial and full refunds work correctly even after the vendor has already withdrawn from their Stripe balance.

```typescript
async function captureCorporateBooking(bookingId) {
  const transaction = await mongoDB.Bookings.findById(bookingId);
  const provider = await mongoDB.Vendors.findById(transaction.vendorId);

  const applicationFee = calculatePlatformPercentage(transaction.amount, provider.tier);

  const paymentIntent = await stripe.paymentIntents.capture(transaction.preAuthId, {
    amount_to_capture: transaction.amount,
    application_fee_amount: applicationFee,
    transfer_data: { destination: provider.stripeAccountId },
  });

  return paymentIntent;
}
```

### Timezone-Aware Recurrence

Vendor availability spans multiple time zones. The scheduler accounts for daylight savings and UTC offsets when generating 60-day recurring availability slots.

### Lifecycle Automation via Hatchet

**Hatchet** workflows replaced fragile cron jobs for:

- Auto-cancelling stale pending bookings
- Voiding expired Stripe pre-authorizations
- No dropped state mutations during booking expirations

## Impact

- **Correct international split payments** — Three-way Stripe Connect splits across vendors, platform, and tax
- **Zero dropped expirations** — Hatchet state machines vs. cron loops = no missed booking cancellations
- **DST-safe scheduling** — Vendor availability correctly shifts with daylight savings across all time zones
