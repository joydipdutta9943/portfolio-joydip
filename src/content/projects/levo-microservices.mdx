> TypeScript for business logic, Go for analytics ingestion, ClickHouse for aggregation — each workload on its optimal runtime.

## The Challenge

A monolithic Node.js backend can't safely ingest millions of raw analytic events while handling ACID-compliant mutations (user signups, payments) on the same event loop. The system needed to separate these workloads without fragmenting the developer experience.

## Key Engineering Decisions

### Hybrid Persistence Layer

Each data type routes to the database built for it:

- **High-volume telemetry** → Go (Fiber) service streaming into **ClickHouse** distributed tables
- **Variable/unstructured data** → **MongoDB** with flexible schemas
- **Core business logic** → TypeScript (Bun) over **PostgreSQL** with strict typing

### Dynamic Query Builder in Go

A translation engine that converts deeply nested JSON filters from the frontend into optimized ClickHouse SQL on the fly. Enterprise operators use this for complex telemetry reporting.

```go
func BuildClickHouseQuery(filters []FilterNode) (string, []interface{}) {
    query := squirrel.Select("user_id, count(*) as triggers").
        From("events_local")

    for _, f := range filters {
        if f.Operator == "in" {
            query = query.Where(squirrel.Eq{f.Field: f.Values})
        } else if f.Operator == "gt" {
            query = query.Where(squirrel.Gt{f.Field: f.Value})
        }
    }

    return query.ToSql()
}
```

### Centralized Worker Pattern

Background job consumers from all Go and TypeScript services consolidated into a single **AWS SQS** deployment. Isolated failure domains with observable dead-letter queues.

## Impact

- **Complete workload isolation** — Analytics ingestion never impacts REST API latency
- **Near real-time reporting** — Dynamic Go query builder delivers aggregate reports in milliseconds
- **Unified observability** — Single SQS deployment with dead-letter queues across all services
